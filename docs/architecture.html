项目架构文档

3DVideo_DEMO 是一个具备 3D 视差渲染、头部追踪、AI 对话与 TTS 语音合成的互动视频展示系统。系统分为三个前端模式：编辑器 (index.html)、查看器 (viewer.html) 与最终产品 (final.html)，共享同一个前端入口 src/app.js，由 server.js 提供 HTTP 服务。

技术栈

后端
  • Node.js 原生 http 模块
  • 原生 fs、zlib、crypto
  • 内置 WebSocket 客户端（讯飞 TTS）
  • 无任何 npm 依赖
前端
  • 原生 Canvas 2D（透视投影）
  • Web Speech API（ASR）
  • MediaPipe FaceLandmarker（头部追踪）
  • 浏览器 ImageDecoder API（GIF 帧控制）
外部服务
  • Moonshot / Kimi AI（对话生成）
  • 讯飞超拟人 TTS（语音合成）
  • PM2（进程管理，ecosystem.config.js）

后端架构 (server.js)

单文件纯 Node.js HTTP 服务器，监听 127.0.0.1:5174，无任何框架。路由用 if-chain 分发，所有处理器均为 async 函数。

数据库读写模式

数据库为单个 JSON 文件 data/scene-layout-db.json，每次读写均通过 readDatabase() 从磁盘加载全量内容，写入时序列化整个对象。

// server.js L738-748
async function readDatabase() {
  await ensureDatabase();
  const raw = await fs.promises.readFile(dbPath, "utf8");  // 每次请求都读磁盘
  const db = JSON.parse(raw);
  return await normalizeDatabase(db);   // 每次都执行完整 normalizeDatabase
}

TTS 流水线（讯飞 WebSocket）

服务器通过 WebSocket 连接讯飞 API，接收乱序音频帧后用 pending Map 按 seq 排序后顺序发送给客户端。

客户端 POST /api/tts → server.js: synthesizeXfyunTtsToStream → WebSocket → 讯飞 API → pending Map 排序 → Chunked audio/mpeg → 客户端

前端架构 (src/app.js)

src/app.js 由三个 HTML 文件共同引用，通过 document.body.dataset.mode ("editor" / "viewer" / "final") 区分运行模式，模式特有功能通过 isViewer / isFinal 标志门控。

状态管理

全局单例 state 对象包含所有运行时状态，无框架或响应式层，变更直接修改属性后手动调用相关更新函数。

  • width, height, dpr — 画布物理尺寸
  • focal — 透视焦距 (520–1300)
  • parallax — 视差强度 (0–180)
  • pointerHead / trackedHead / currentHead — 三路头部位置
  • cameraOn, tracker — 追踪状态
场景与媒体状态
  • items[] — 当前场景图层数组
  • currentSceneId, scenes[] — 场景列表
  • sceneFlow — 场景流转配置
  • gifElements, gifPlayers, gifPauseFrames — GIF 管理 Map
  • paused, scenePlaybackToken — 播放控制
语音状态 (voice)
  • recognition — SpeechRecognition 实例
  • listening, busy — 识别状态机
  • conversation[] — Kimi 对话历史
  • transcriptBuffer, interimTranscript — ASR 输出缓冲
  • micStream, micAudioContext, micAnalyser — 麦克风监控

  • userVariables{} — 运行时变量（如 user_age）

  • activeTts — 当前播放的 TTS 音频元素
  • realtimeReply — 实时回复模式标志

渲染管线

以 requestAnimationFrame 驱动的 60fps Canvas 2D 渲染循环，实现透视投影 + 视差效果。

每帧执行顺序

draw() [60fps rAF loop] ├─ updateHead() — EMA 平滑头部位置 (系数 0.12) ├─ drawBackdrop() — 渐变背景填充 ├─ drawPerspectiveGrid()— 透视网格 (可选) ├─ drawItems() — 每帧按 z 深度排序 → 逐层绘制 │ └─ drawItem(item) — 透视投影 + 变换 + 绘制 ├─ updateGifOverlays() — 同步 DOM GIF/canvas 位置与可见性 └─ drawReticle() — 中心准星

数据流

编辑保存流

用户操作 (拖拽/滑块) → scheduleSaveLayout() (1s 防抖) → POST /api/layout → 写入 scene-layout-db.json

语音交互流

Web Speech API → 中文 ASR → POST /api/director-cue → Kimi AI (Moonshot) → POST /api/tts → 讯飞 WebSocket TTS → audio/mpeg → 播放 + 字幕

头部追踪

支持三种追踪器，按能力依次降级：

1. MediaPipe FaceLandmarker (468 关键点，精度最高) └─ 提取眼角坐标 (landmark 33/133, 362/263) → 计算头部 x/y/z 2. 浏览器 FaceDetector API (原生，精度中等) └─ 边界框中心 + 宽度 → x/y/z 3. Noop Tracker (无摄像头，鼠标指针代替)

追踪结果经 EMA 平滑 (系数 0.12，每帧约 12% 混合) 写入 state.currentHead。

语音管线

ASR（语音识别）

  • 使用 Web Speech API (SpeechRecognition，lang: zh-CN，连续模式)
  • 最短监听时间 minListenMs: 3000ms，避免误触发
  • 停止后自动重启，实现类连续识别

AI 对话（Kimi）

  • 超时 12s (KIMI_TIMEOUT_MS)，最大 160 token (KIMI_MAX_TOKENS)
  • 内置角色 "夏仁"，四幕叙事结构：opening → choice → reveal → ending
  • 超时时返回空 cue + warning，不影响流程

TTS（讯飞超拟人）

  • 4 种声线：lingfeiyi / lingxiaoxuan / lingyuyan / lingbosong
  • WebSocket 协议，15s 超时，按 seq 排序乱序帧
  • 支持流式 (/api/tts-stream) 和完整 (/api/tts) 两种模式

媒体处理

图片 (PNG / GIF)
  • PNG：直接 drawImage 到 canvas
  • GIF 循环：DOM <img> 浮层覆盖 canvas
  • GIF 不循环：ImageDecoder 逐帧解码，末帧暂停
  • 暂停帧缓存在 gifPauseFrames Map
视频 (MOV / WebM)
  • 每帧从 <video> 元素抓帧到 canvas
  • 支持 HTTP Range 请求（视频 seek）
  • WebM 独立循环控制 (webmLoop)
  • 上传大小限制：512 MB

数据库

单个 JSON 文件 data/scene-layout-db.json，结构如下：

{
  "settings": { "finalStartSceneId": "scene-id" },
  "layouts": {
    "default":  { "name": "默认场景", "payload": {...}, "updatedAt": "ISO8601" },
    "scene-2":  { "name": "场景2",    "payload": {...}, "updatedAt": "ISO8601" }
  }
}
每次 readDatabase() 都从磁盘读取全量 JSON 并执行 normalizeDatabase()，当场景数量增多时会造成明显 I/O 和 CPU 开销。

后端性能优化建议

#问题位置影响建议方案
B-1 数据库无内存缓存：每次 API 请求均从磁盘读取全量 JSON server.js L738 高 引入模块级 let dbCache = null，写入后更新缓存；脏标记驱动持久化。可用 debounce 批量合并写操作。
B-2 normalizeDatabase 每次全量执行：读取时无论是否变化均完整遍历 server.js L750 中 B-1 缓存引入后 normalizeDatabase 仅在写入时运行一次，读取直接返回缓存对象。
B-3 Gzip 压缩未缓存：每次请求静态文件都即时压缩，对大型 JS 文件影响显著 server.js L597 中 启动时预压缩 HTML/JS/CSS 文件到内存 Map，请求时直接返回。或在文件变更时失效缓存（用 mtime 校验）。
B-4 无 ETag / If-None-Match：浏览器无法做条件请求，每次都下载完整静态文件 server.js L532 中 在 serveStaticFile 中用文件 mtime + size 生成弱 ETag，添加 ETag 响应头并处理 If-None-Match，命中时返回 304。
B-5 资产列表每次扫描磁盘：GET /api/assets 实时遍历 mat/ 和 uploads/ 目录 server.js L813 低 维护内存资产列表，在上传操作 (POST /api/assets) 后增量更新，避免 GET 时扫目录。
B-6 JSON 写入非原子操作：直接 writeFile(dbPath) 可能在崩溃时产生损坏文件 server.js L704 低 写入临时文件 dbPath + ".tmp"，完成后 fs.rename()（原子替换）。加写互斥锁防并发写冲突。
B-7 scriptBeats 前后端重复定义：内容相同的对象在服务端和客户端各维护一份 server.js L55 / app.js L147 低 通过 GET /api/script-beats 接口暴露，前端动态拉取。或将共享配置提取到 data/script-beats.json。

B-1 数据库缓存示例

// server.js — 在 readDatabase / writeLayoutPayload 修改处

let dbCache = null;   // 内存缓存
let dbWriteTimer = null;

async function readDatabase() {
  if (dbCache) return dbCache;           // 命中缓存，跳过磁盘读取
  await ensureDatabase();
  const raw = await fs.promises.readFile(dbPath, "utf8");
  dbCache = await normalizeDatabase(JSON.parse(raw));
  return dbCache;
}

async function persistDatabase() {
  if (!dbCache) return;
  const tmp = dbPath + ".tmp";
  await fs.promises.writeFile(tmp, JSON.stringify(dbCache, null, 2), "utf8");
  await fs.promises.rename(tmp, dbPath);  // 原子替换 (B-6)
}

// 防抖写入：500ms 内多次变更合并为一次磁盘操作
function schedulePersist() {
  clearTimeout(dbWriteTimer);
  dbWriteTimer = setTimeout(persistDatabase, 500);
}

async function writeLayoutPayload(id, name, payload) {
  const db = await readDatabase();
  db.layouts[id] = { name, payload: { ...payload, id, name }, updatedAt: new Date().toISOString() };
  schedulePersist();  // 替换原来的 fs.promises.writeFile
}

前端性能优化建议

#问题位置影响建议方案
F-1 rAF 循环在 paused/后台不停歇：页面不可见或暂停时仍以 60fps 运行 app.js L194 高 使用 document.addEventListener("visibilitychange") + Page Visibility API：页面隐藏时暂停 rAF，重新可见时恢复。场景暂停且无追踪时也可降帧（setTimeout 8fps 代替 rAF）。
F-2 items 每帧排序：drawItems() 在每个 rAF 帧对图层数组按 z 深度排序 app.js drawItems() 中 维护一个预排序的 sortedItems 数组，仅在图层 z 值变化时重新排序（脏标记）。拖拽时图层数量有限，实际 sort 成本低，但可消除 final 模式下无操作时的多余排序。
F-3 resize 无节流：窗口 resize 事件直接触发 canvas 尺寸重算 app.js L421 低 用 ResizeObserver 替代 window.resize 事件，或加 16ms debounce。ResizeObserver 更精准且不会在无变化时触发。
F-4 拖拽时重建图层列表 DOM：pointermove 拖拽触发 renderLayerList()，每次重建全部 DOM 节点 app.js L369 中 拖拽期间跳过 renderLayerList()，仅更新坐标显示值；pointerup 结束时再做一次完整刷新。
F-5 人脸检测每帧运行：MediaPipe/FaceDetector 在每个 rAF 帧都调用推理 app.js tracking loop 中 将追踪帧率降至 15–20fps（每 3–4 帧推理一次），渲染仍保持 60fps，利用上帧 EMA 值插值。EMA 系数 0.12 的平滑性可掩盖追踪帧率降低带来的抖动。
F-6 GIF 状态管理用三个 Map：gifElements、gifPlayers、gifPauseFrames 分散管理，同一图层数据分布在三处 app.js L120-122 低 合并为单一 Map，值为对象 { element, player, pauseFrame }，降低查找复杂度和维护成本。
F-7 layoutSaveTimer 每次变更完整序列化：拖拽中 1s 内 scheduleSaveLayout 多次触发都会在到期时完整 POST 场景数据 app.js L98 低 当前防抖已是 1s，实际影响不大。如需进一步优化，可在拖拽结束 (pointerup) 时才触发保存，拖拽中只更新本地 state。

F-1 Page Visibility 示例

// src/app.js — 在 init() 后添加

let rafId = null;

function startDrawLoop() {
  if (rafId) return;
  function loop() {
    draw();
    rafId = requestAnimationFrame(loop);
  }
  rafId = requestAnimationFrame(loop);
}

function stopDrawLoop() {
  if (rafId) cancelAnimationFrame(rafId);
  rafId = null;
}

document.addEventListener("visibilitychange", () => {
  if (document.hidden) stopDrawLoop();
  else startDrawLoop();
});

F-5 追踪帧率节流示例

// 在 tracking runDetection 循环中添加节流

let trackFrameCount = 0;
const TRACK_EVERY_N_FRAMES = 3;  // 60fps ÷ 3 = 20fps 推理

function trackingLoop() {
  trackFrameCount++;
  if (trackFrameCount % TRACK_EVERY_N_FRAMES === 0) {
    runFaceDetection();  // 仅每 3 帧执行一次推理
  }
  requestAnimationFrame(trackingLoop);
}

架构级优化建议

#问题影响建议方案
A-1 credentials 硬编码在 ecosystem.config.js：讯飞/Moonshot 密钥出现在 PM2 配置文件中，易被意外提交 高（安全） 从 ecosystem.config.js 删除所有 secrets，改从 .env 文件或 OS 环境变量注入。在 .gitignore 中确认 .env 已被忽略。
A-2 单文件 HTTP 服务器缺乏并发保护：多个请求同时写 DB 时存在竞态，无请求排队 中 引入写互斥锁（简单 Promise 队列）保护 writeLayoutPayload 和 writeSettings，或结合 B-1 缓存方案中的 schedulePersist 批量写。
A-3 TTS 无请求去重/缓存：相同文本重复 TTS 请求每次都经过讯飞 WebSocket，浪费 RTT 中 用 Map<text+voice, Promise<Buffer>> 做进行中请求去重（deduplication），相同文本的并发请求共享同一个 Promise。可选：对频繁使用的短语做内存缓存，TTL 5 分钟。
A-4 前端错误无上报机制：AI 超时、TTS 失败等错误仅 console 输出，运营无法感知 低 添加简单的 POST /api/log-error 接口，前端在关键失败路径上调用，服务端写 append-only 日志文件。
A-5 场景切换无预加载："auto" / "dialog" 模式切换下一场景时才开始加载媒体，有明显延迟 中 在当前场景媒体播放中期（如 50% 进度）异步预加载 nextSceneId 的资源到内存，场景切换时直接应用，消除切换延迟。
A-1 安全优先：在 ecosystem.config.js 中发现 API keys 硬编码风险。部署到生产环境前必须迁移到环境变量，并确认密钥未被提交到 git 历史。
